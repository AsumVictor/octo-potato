Abstract

Campus navigation and facility issue reporting are two recurring operational challenges at Ashesi University. Students, staff, and visitors struggle to locate destinations within the campus due to its winding layout, while the existing mechanism for reporting maintenance issues relies on informal email chains that offer limited spatial precision.

This paper presents the design, development, and evaluation of an interactive 360° panorama navigation and facility issue reporting system built for Ashesi University. The system embeds a Pano2VR-rendered equirectangular panorama tour of the campus into a web-based application and augments it with a custom vanilla JavaScript navigation layer that provides Dijkstra-based shortest-path routing, live GPS positioning via the browser's Geolocation API, and gyroscope- driven look-around on mobile devices.

A key contribution of the system is its spatially-aware issue reporting module. Users draw a bounding rectangle directly on the panorama viewport to mark the exact location of a campus problem, attach photographs, and submit a structured report to a Supabase PostgreSQL backend. A companion logistics view allows facilities staff to open any submitted report and be automatically navigated to the precise panorama node and camera angle from which the report was filed, with the original selection rectangle re-drawn as a visual overlay.

The system is implemented entirely in client-side JavaScript with no build toolchain, making it straightforward to drop into any Pano2VR export and redeploy after a new campus scan.

Chapter 1: Introduction

1.1 Background

Ashesi University occupies a hilly campus in Berekuso, Ghana, where winding pathways connect academic buildings, residence halls, dining facilities, and support services. New students, visitors, and prospective applicants regularly struggle to locate destinations, and no dedicated digital wayfinding tool existed prior to this work. In parallel, the facilities management workflow relied on email submissions to a central support address, a process that lacks spatial context and makes it difficult for logistics staff to understand precisely where on campus a fault has occurred [1].

Advances in consumer 360° cameras and photogrammetry software now make it feasible to produce high-quality virtual tours of campus environments at relatively low cost. The Pano2VR authoring platform, in particular, allows a photographer to link equirectangular panorama images into a navigable graph of hotspot-connected nodes and export a self-contained HTML5 application. This provides the rendering substrate on which the present system is built.

1.2 Problem Statement

Two discrete but related problems motivate this project. First, there is no interactive digital map or panoramic wayfinding tool that allows an Ashesi campus user to search for a destination and receive turn-by-turn guidance through the physical environment. Second, the existing fault-reporting mechanism provides no spatial anchor for reports, making it harder for facilities staff to prioritise and locate problems quickly. A unified solution that addresses both problems within a single, low-maintenance web application is therefore desirable.

1.3 Related Works

Armo-Himbson (2020) built a facilities management system for Ashesi University using three-dimensional building models rendered with three.js. The system allowed users to click on 3D objects to file reports and gave administrators a dashboard for tracking and assigning remediation tasks [2]. The present work extends this concept by replacing static 3D models with photorealistic 360° panoramas and adding active navigation as a first-class feature.

Google Street View demonstrates large-scale deployment of navigable equirectangular panoramas linked by a graph structure, supporting millions of nodes worldwide. While the underlying technology is comparable, Street View does not expose pathfinding or facility reporting APIs, and its data is not current enough for indoor or private-campus use.

Indoor navigation research by Afyouni et al. (2012) established graph-based shortest-path routing as the standard approach for wayfinding in environments where GPS is unavailable or inaccurate [3]. Their work informs the adjacency graph model and Dijkstra implementation used in this system.

Chapter 2: Requirements Specification

2.1 Data Collection

Semi-structured interviews were conducted with three groups: (i) first-year students unfamiliar with the campus layout, (ii) members of the Logistics and Facilities team, and (iii) two members of the IT support centre. Key findings were that students wanted a search-based tool rather than a static map, facilities staff wanted reports to carry location photographs and precise positional metadata, and the IT team required a solution requiring no server-side runtime to minimise hosting complexity.

2.2 Functional Requirements

  1. Users shall search for a campus location by name and receive shortest-path navigation guidance through the panorama.
  2. The system shall support live GPS positioning to determine the user's nearest panorama node automatically.
  3. Users shall draw a rectangular selection on the panorama viewport to spatially anchor an issue report.
  4. Reports shall capture the panorama node identifier, camera pan/tilt/fov, a normalised viewport rectangle, descriptive text, issue category, and optional photographs.
  5. Reports shall be persisted to a cloud database and be retrievable by facilities staff via a logistics URL.
  6. The logistics view shall navigate the panorama to the exact node and camera angle of any submitted report and redraw the original selection rectangle.

2.3 Non-Functional Requirements

  1. Usability: The navigation and reporting workflows shall be completable by a first-time user without instructions within three minutes.
  2. Performance: Initial tile preloading shall complete within ten seconds on a standard campus Wi-Fi connection.
  3. Portability: The system shall be deployable by dropping a set of JavaScript files into any Pano2VR-generated export folder with no build step.
  4. Availability: The cloud backend (Supabase) shall provide 99.9% uptime for report submission and retrieval.

Chapter 3: Design and Architecture

3.1 System Overview

The system is structured as a layered client-side application built on top of the Pano2VR- generated HTML5 panorama player. All custom code is organized into modules using the Immediately Invoked Function Expression (IIFE) pattern, sharing a single global namespace window.Nav. There is no framework dependency or bundler; each module file is a self-contained script tag. A single App.js boot sequence wires all modules together after the Pano2VR player fires its configloaded event.

3.2 Module Architecture

The codebase is divided into six functional layers:

  1. Core — EventBus (pub/sub message bus) and AppState (shared runtime state).
  2. Data — XmlLoader, NodeParser, GraphBuilder, Preloader, IssueTypes, SupabaseClient, IssueService, LogisticsLoader.
  3. Pathfinding — Pathfinder implements Dijkstra's algorithm on the weighted adjacency graph produced from the pano.xml hotspot data, using Haversine great-circle distance as the edge weight where GPS coordinates are available.
  4. Navigation — Navigator steps through the computed path one node at a time, and AutoRotator aims the camera toward the next hotspot during idle periods.
  5. Rendering — Renderer and Projector draw the canvas-based directional arrow and glowing ring that indicate the next waypoint on screen.
  6. UI — StyleInjector, UIBuilder, SearchPanel, ModeChooser, ReportTool, IssueViewer, HUD, Toast, LoadingOverlay.

3.3 Database Design

The backend uses Supabase (PostgreSQL) with Row Level Security. Four tables are defined:

  • issue_types — Reference table of issue categories (safety, fire, damage, flooding, etc.) seeded at setup; drives the report form dropdown at runtime.
  • issue_statuses — Workflow states: open, in_progress, resolved, closed, rejected, pending. New reports default to open via a PostgreSQL column default function.
  • issues — Core table storing reporter email, issue type FK, status FK, a JSONB metadata column (reporter name, description, severity, panorama location), and a TEXT[] images array of Supabase Storage URLs.
  • resolves — Records resolution events linking an issue to an authenticated staff member.

Only Ashesi email addresses (@ashesi.edu.gh, @ug.ashesi.edu.gh) are accepted for issue submission, enforced by both a PostgreSQL CHECK constraint and an RLS INSERT policy.

Chapter 4: Implementation

4.1 Panorama Navigation

At boot, XmlLoader fetches the Pano2VR-generated pano.xml file and NodeParser extracts node identifiers, titles, GPS coordinates, and hotspot links. GraphBuilder constructs a weighted adjacency list and performs a connectivity check that logs isolated nodes — a common symptom of broken hotspot links in Pano2VR exports.

When the user selects a destination, Pathfinder runs Dijkstra's algorithm and returns an ordered path array. Navigator listens to the Pano2VR changenode event and advances the step index on each transition. Renderer projects the next waypoint from equirectangular coordinates to screen space on every animation frame and draws a pulsing ring with a directional bounce arrow using the HTML5 Canvas API.

4.2 Live GPS Mode

When the user selects Live Location, LiveController calls navigator.geolocation.watchPosition. On each GPS update, the system computes the Haversine distance from the device's coordinates to every node in the graph and navigates the panorama to the nearest node automatically.

4.3 Issue Reporting

The report button activates a transparent canvas overlay with a crosshair cursor. The user drags a rectangle, which is rendered with a dashed stroke using setLineDash. On mouse-up, the system records the canvas-space bounding box, normalises it to a 0–1 viewport fraction, and captures the Pano2VR camera state (pan, tilt, FOV) and current node metadata. The report dialog then opens, pre-populated with the issue categories fetched from Supabase at boot.

On submission, IssueService uploads attached images to the issue-images Supabase Storage bucket (sanitising filenames to remove spaces and special characters), then inserts the issue record. The status_id column is set automatically by a PostgreSQL helper function (open_status_id()), eliminating a client-side lookup round-trip that previously caused failures under restrictive Row Level Security configurations.

4.4 Logistics View

The companion page logistics.html?id=<uuid> fetches the issue record from Supabase (joining issue_types for the category label in a single query), displays a structured 320px side panel with all report metadata, and calls pano.openNext() with the stored node ID and camera angles to reproduce the original viewing position. The stored normalised viewport rectangle is then re-projected onto a canvas overlay as an orange dashed selection box, and a Go to Location button allows staff to re-trigger the navigation and re-draw the highlight after panning away.

Chapter 6: Conclusion

6.1 Overview

This project set out to address two campus challenges at Ashesi University — the absence of interactive wayfinding and the lack of spatial precision in facility fault reporting — through a single, lightweight, browser-based application. Both objectives were achieved. The system provides shortest-path navigation through a photorealistic panorama tour, supports live GPS positioning for hands-free guidance, and allows users to file spatially-anchored issue reports that facilities staff can review in context through the companion logistics view.

A deliberate architectural choice was made to avoid build toolchains and framework dependencies. This keeps deployment friction near zero: the system is added to a Pano2VR export by copying a folder of scripts and adding script tags to index.html. After any campus re-scan, only the pano.xml and tile files need to be replaced.

6.2 Challenges

Pano2VR API stability. The player API exposes no formal specification, so methods such as getPan(), setView(), and openNext() were verified empirically. Future Pano2VR version upgrades may require corresponding updates to the navigation layer.

Supabase Row Level Security. Multiple submission failures during development were traced to RLS silently returning zero rows for the anon role on reference tables that lacked SELECT policies. This was resolved by adding permissive SELECT policies to issue_types and issue_statuses, and by eliminating the per-submit issue_statuses lookup entirely in favour of a PostgreSQL column default function.

GPS accuracy on campus. The Geolocation API returns accuracy radii between 5 m and 40 m depending on device and environment. In dense indoor areas, the nearest-node algorithm occasionally snaps to an adjacent building. A fallback to manual mode is provided for these cases.

Panorama tile size. The full tile set for the campus tour exceeds 2 GB, making the repository large and deployment slow. Migration to a CDN (Cloudflare R2 or equivalent) is identified as a priority for production deployment.

6.3 Future Work

  • CDN tile hosting. Migrating panorama tiles to Cloudflare R2 would reduce repository size and improve global load times, with zero egress fees for public access.
  • Administrator dashboard. A web interface for the Logistics team to triage, assign, and update the status of submitted issues would close the workflow loop opened by the reporting module.
  • Indoor node GPS correction. A BLE beacon or Wi-Fi fingerprinting layer could supplement GPS accuracy within buildings.
  • Accessibility. Keyboard navigation support and ARIA labelling for the custom Canvas elements would improve usability for users with visual or motor impairments.
  • Multi-campus support. The module architecture is campus-agnostic; parametrising the Supabase project and pano.xml path would allow multiple institutions to run independent instances from the same codebase.

6.4 Conclusion

The 360° Campus Navigation and Issue Reporting System demonstrates that a photorealistic, graph-navigated panorama tour can serve as an effective substrate for both wayfinding and spatially-precise facility reporting. By building directly on a Pano2VR export and integrating with a serverless Supabase backend, the project achieves its design goals with no server runtime and minimal maintenance overhead. The work extends the earlier facilities management contribution by Armo-Himbson (2020) and establishes a foundation for a fully digital campus facilities lifecycle at Ashesi University.
